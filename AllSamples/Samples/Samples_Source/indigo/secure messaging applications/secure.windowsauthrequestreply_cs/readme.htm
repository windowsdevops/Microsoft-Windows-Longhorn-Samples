Overview

The "Longhorn" communications subsystem (code-named "Indigo") can be used to help secure applications. The WindowsAuthRequestReply sample is a basic application consisting of a client, a service, a service host, and client and host configuration and data files and using Windows (Kerberos) authentication. It demonstrates the simplicity how to add Windows security to a service framework application. While the default behavior of the "Indigo" security infrastructure sets the client and server to use Windows security and by default all methods are secured using this model, this sample explicitly sets security on the methods for demonstration purposes. The client is a console application that sends a message secured using the current thread's Windows token to the server. The message body is a simple string. The server is a console application that validates the Kerberos token and formulates a response using the string from the message.

Key Concepts

Client:
The client is a simple console application that uses a configuration file to store settings for the objects that it uses.

    
<configuration>
  <system.messagebus>
    <serviceEnvironments>
      <serviceEnvironment name="main">
        <port>
            <identityRole>soap.tcp://localhost:46001/HelloClient/</identityRole>
        </port>
        <!-- WARNING: This sample does not secure the server policy for demonstration purposes. -->
        <policyManager>
          <!-- Permits untrusted policy statements. Default requires signed policy statements. -->
          <areUntrustedPolicyAttachmentsAccepted>true</areUntrustedPolicyAttachmentsAccepted>
          <isPolicyReturned>true</isPolicyReturned>
        </policyManager>
      </serviceEnvironment>
    </serviceEnvironments>
  </system.messagebus>
</configuration>
 

The client loads this configuration information, uses the service manager to create a proxy that represents a service, and invokes the method on the proxy. Because this service method is an RPC-style or request-reply style method, the execution of the client stops until the call returns.

It is important to realize that although this client sends SOAP 1.2 messages to communicate with the service, you interact with this service in your client as though it were a local call and used no custom parameter types. Note that there is no explicit setting to indicate that you are using Windows credentials. That information is stored in the Machine.config file, which makes it easy to develop a secure application that relies on default settings. 

C# client:
string name = "Client";
if (args.Length > 0)
name = args[0];
 
// Load the default service environment, called "main".
ServiceEnvironment se = ServiceEnvironment.Load();
 
// Retrieve the ServiceManager object from the default environment
ServiceManager manager = se[typeof(ServiceManager)] as ServiceManager;
if (manager == null)
{
  throw new ApplicationException(
  "The ServiceManager is not available in the service se.");
}
// Start the service environment.
se.Open();
 
// Create a proxy channel that points to the service to call.
Uri uri = new Uri("soap.tcp://localhost:46000/HelloService/");
IHelloChannel channel = (IHelloChannel)manager.CreateChannel(typeof(IHelloChannel), uri);
 
try 
{
  Console.WriteLine(channel.Greeting(name));
}
catch(Exception Ex)
{
  Console.WriteLine(Ex);
}
finally
{
  se.Close();
}

Securing the service:

1. Service:

The service itself is marked with the DatagramPortTypeAttribute as before and the methods are marked with the ServiceMethodAttribute. In addition, the service has a new  attribute, named ServiceSecurity(), that indicates that the method is to be secured. This establishes a security scope such that all messages matching the scope will require the constraints it establishes. By default this security scope is bound to a default profile that specifies the use of Windows authentication. That means all the users in the current domain will be allowed to access this method. (You can also constrain the list of users allowed access to the method further, but that is not demonstrated in this sample.  By the time the Greeting method is invoked, all relevant authentication and access checking for "mbse:StandardUserRole" has been performed.

C# Service:

    [DatagramPortType(Name="Hello",Namespace="http://www.tempuri.org/quickstarts")] 
    public class Hello 
    { 
     [ServiceSecurity(Name = "StandardScope ", Role = "StandardUserRole")]
        [ServiceMethod] 
        public string Greeting(string name) 
        { 
            Console.WriteLine("Called by a client with the name {0}", name); 
            return String.Format("Hello, {0}!", name); 
        } 
    }

VB Service:

   'DatagramPortType attribute indicates a sessionless service

   <DatagramPortType(Name:="Hello", Namespace:="http://www.tempuri.org/quickstarts")> _

   Public Class Hello

 

       ' ServiceMethod attribute exposes methods for access on the service

       <ServiceSecurity(Name:="StandardScope", Role:="StandardUserRole")> _

       <ServiceMethod()> _

                           Public Function Greeting(ByVal name As String) As String

           Console.WriteLine("Called by a client with the name {0}", name)

           Return String.Format("Hello, {0}!", name)

       End Function

   End Class

2. Host:
The host application configures "Indigo" to listen for incoming messages for the services listed in the <activatableServices> element. In addition, the host application specifies that the Windows profile be used for binding to the scope. The HelloService method code defines the standardscope scope using the ServiceSecurityAttribute.

 
<configuration>
  <system.messagebus>
    <serviceEnvironments>
      <serviceEnvironment name="main">
        <port>
          <identityRole>soap.tcp://localhost:46000/HelloService/</identityRole>
        </port>
        <!-- Bind the scope defined using ServiceSecurityAttribute to a profile that uses Windows authentication. -->
        <securityManager>
               <applicationSecurity>
                       <binding scope="StandardScope " profile="windows" />
               </applicationSecurity>
        </securityManager>
        <!-- WARNING: Policy is not secured to clarify demonstration. -->
        <policyManager>
          <!-- Permits unsigned policy statements. The default requires signed policy statements. -->
          <areUntrustedPolicyAttachmentsAccepted>true</areUntrustedPolicyAttachmentsAccepted>
          <isPolicyReturned>true</isPolicyReturned>
        </policyManager>
        <serviceManager>
          <activatableServices>
            <add type="Hello, HelloService" />
          </activatableServices>
        </serviceManager>
      </serviceEnvironment>
    </serviceEnvironments>
  </system.messagebus>
</configuration>

 

3. Data file:
<securityData>
    <authorizationData>
        <memoryMapping id="mainAuthorizationData">
            <globalEntries>
                <windowsRoleAssignment domain="MyDomain" group="Domain Users" roles = "StandardUserRole"/>
            </globalEntries>
        </memoryMapping>
    </authorizationData>
</securityData>
 

The data file shows the mapping for authorization data: all users in the MyDomain\"Domain Users" group are mapped to the StandardUserRoles role. This role is allowed access to the Greeting method (as specified in the code).

Building

  1. Open the Longhorn Build Environment console window (click Start, click All Programs, click Microsoft Longhorn SDK, click Open Build Environment Window, and click Longhorn Build Environment).
  2. Navigate to the directory where this readme is located.
  3. Type MSBuild and press ENTER.

Running

To run the sample, you must change the "MyDomain" value in the Host.exe.security file to a valid domain name.  

Running on a single machine:
To run the sample on a single machine, you must change the "MyDomain" value in the Host.exe.security file to a valid domain name.  Next, open a command prompt window in the Server directory and another one in the Client directory. At the server command prompt, type host to start the server application, Then, at the client directory console prompt, type client <name> to start the client application.

Running on multiple machines:
To run the sample on multiple machines, perform the following steps:

  1. On the server machine, edit the Host.exe.security file in the Server directory.  Replace "MyDomain" with a valid domain name.
  2. On the server machine, edit the Host.exe.config file in the Server directory.. Replace "localhost" with the server machine name. Then run the server application.
  3. On the client machine, edit the Client.cs (or Client.vb) file in the Client directory. In the URI that represents the server location, replace "localhost" with the server machine name. Recompile the client and rerun the client application.
  4. On the client machine, edit Client.exe.config.  Replace "localhost" in the <identityRole> element with the client machine name.

Note:

To run this sample, your computer must be attached to a Microsoft Windows domain.  If your computer is attached only to a workgroup, this sample will fail.
